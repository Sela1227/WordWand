## 一、用 Kit 的整體感受

第一個「前端靜態 + 後端代理」雙端混合專案。Kit 靜態頁示範與 FastAPI 經驗各自夠用,但兩者**併在同一 zip / 同一 repo 的打包結構**(docs/ 給 Pages、backend/ 給 Railway)沒有現成 reference,是這次自己決定的。配色「兒童依向性自訂」與「不套 logo(用戶覆蓋)」兩條規則都有明確指引可循,沒有卡點。

## 二、跨專案通用坑建議

> 已 grep Kit,以下為「可能可回流」候選,皆 **N=1**,建議先觀察、不急著進坑庫。

1. **[N=1] 開發者自有 key 服務多名公開使用者 → 必須後端代理藏 key**
   - 跟 `sela-philosophy 一.5`「用戶自己的 key 存 localStorage」是**對立情境**:那條講「使用者各自的 key」,這條講「SELA 自己的 key 服務一群不擁有 key 的使用者(如學生)」,後者絕不能放前端,必須走後端代理 + 伺服器端組 prompt 限制用途。
   - 建議:N≥2(下個 AI 公開服務專案)再考慮在 sela-philosophy 補一條對照。

2. **[N=1 新 stack 訊號] React via CDN + Babel standalone(零建置)**
   - 用於極小型互動工具,免 Vite/CRA build、Git Pusher 推一份即上 Pages。
   - 屬「既有 framework 的新部署方式」(方向 2),沒踩坑,僅標訊號。N≥2 再評估是否值得開 reference。

## 三、設計模式建議(只取會改變決策方式的)

- **「前端只顯示、人格 prompt 放後端」的職責切割**:前端 `SPIRITS` 只管外觀(名稱/配色/介紹),語氣 `PERSONAS` 放後端。好處是改 AI 行為不動前端、也防使用者竄改 prompt 盜用 key。這個「把可被濫用的邏輯收進後端」原則對任何 AI 公開服務通用,但仍 N=1,建議觀察。
- **[N=1] 兒童 / 受限對象 AI 服務的「範圍 + 內容」雙閘把關模式**(V0.2.0 新增):安全規則放後端 prompt 最前面(最優先)、由模型回 `ok` 旗標分流、後端再做 fail-safe(非明確通過一律擋)。前端絕不負責把關(可繞過)。對任何「給特定族群、需限制可問範圍 / 需內容分級」的 AI 服務通用(兒童教育、客服機器人、單一用途工具皆是)。N=1,建議下個受限對象 AI 專案再評估進 Kit。
